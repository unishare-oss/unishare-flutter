    SPEC-0011: Edit and Delete Own Post¶

    Status: APPROVED Author: architect Date: 2026-05-19 Proposal: PROP-0011 Approved by: Slade

    Overview¶

    Post authors need a way to correct mistakes and remove their own content. This spec adds a PopupMenuButton to the PostDetailScreen AppBar that is visible only when post.authorId == currentUser.uid. From that menu, the author can either navigate to an EditPostScreen (allowing changes to title, description, tags, externalUrl, and moduleNumber) or trigger a hard delete that removes the Firestore document and all associated Storage files listed in post.mediaUrls and post.codeSnippetUrl. The AI summary fields are automatically reset to pending when description changes so the worker can re-trigger. All other post fields — authorId, courseId, year, semester, postingIdentity, mediaUrls, mediaTypes, likesCount, createdAt, summaryStatus source-of-truth in Storage, summarizedAt — remain immutable from the client.

    Firestore rules¶

    The existing posts/{postId} block currently has: - allow read: if true; - allow create guarded by authorId == uid and title non-empty. - allow update limited to likesCount (any auth user) and summaryStatus/summary/summarizedAt (author only, via partial-update path). - allow delete: if false;

    Replace the allow update and allow delete clauses inside match /posts/{postId} with the following:

    allow delete: if request.auth != null
              && request.auth.uid == resource.data.authorId;

allow update: if request.auth != null
              && (
                // Any authenticated user may increment/decrement likesCount.
                request.resource.data.diff(resource.data).affectedKeys()
                  .hasOnly(['likesCount'])
                ||
                // Owner may edit mutable content fields.
                (
                  request.auth.uid == resource.data.authorId
                  && request.resource.data.diff(resource.data).affectedKeys()
                       .hasOnly([
                         'title', 'description', 'tags', 'externalUrl',
                         'moduleNumber', 'updatedAt',
                         'summary', 'summaryStatus', 'summarizedAt'
                       ])
                  && request.resource.data.title is string
                  && request.resource.data.title.size() > 0
                )
                ||
                // Worker-only summary update path (same as before — author owns doc).
                (
                  request.auth.uid == resource.data.authorId
                  && request.resource.data.diff(resource.data).affectedKeys()
                       .hasOnly(['summaryStatus', 'summary', 'summarizedAt'])
                )
              );

    Why the allowlist matters: Firestore's hasOnly constraint prevents a malicious client from sneaking in writes to immutable fields (authorId, courseId, year, semester, postingIdentity, mediaUrls, mediaTypes, likesCount, createdAt) inside an otherwise valid edit request. The summary fields (summary, summaryStatus, summarizedAt) are included in the owner-edit allowlist because UpdatePost must be able to reset them to pending atomically in the same write when description changes.

    Deletion sequence¶

    PostRepositoryImpl.deletePost must follow this order to avoid orphaning Storage objects if the Firestore delete fails:

    1. For each URL in post.mediaUrls: call storageDatasource.deleteFile(url). Errors are swallowed individually (best-effort) so a single missing file does not abort the whole operation.
    2. Call storageDatasource.deleteFile(post.codeSnippetUrl) (no-op if null).
    3. Call firestoreDatasource.deletePost(postId).

    The Firestore delete is last because watchPost drives the UI. Once the document is deleted, PostDetailScreen receives a StateError('post_not_found'), renders the inline not-found state, and the user is unblocked. Storage orphan risk is lower than a Firestore document existing with broken media references — hence Storage first, Firestore last.

    EditPostScreen behaviour¶

    • Accepts postId: String and post: Post? (nullable seed from route extra).
    • If post is null (direct deep-link entry), the screen calls watchPost(postId) to hydrate — renders a loading indicator until the stream emits.
    • Pre-populates title, description, tags, moduleNumber, externalUrl from the Post entity.
    • Computes descriptionChanged by comparing the submitted description to the original at the time of submission.
    • On save: calls editPostProvider.save(...). On AsyncData, calls context.pop(). On AsyncError, shows a SnackBar with the error message.
    • Blocked fields (courseId, year, semester, postingIdentity, mediaUrls) are displayed read-only for context but excluded from the submitted payload.
    • Tag editing: the existing TagInputField widget (or equivalent chip-input pattern) from CreatePostScreen must be reused — no new tag input widget.

    Out of scope¶

    • Editing mediaUrls or mediaTypes — file re-upload is a separate, higher-complexity feature.
    • Editing codeSnippetUrl — same rationale as media files.
    • Editing courseId, year, semester, or postingIdentity — these are structural classification fields; changing them would require re-indexing and feed cache invalidation beyond the scope of a simple edit.
    • An "Edited" indicator in PostDetailScreen (e.g. showing "edited" when updatedAt != createdAt) — deferred to a follow-up spec.
    • Soft delete or recycle-bin behaviour — post deletion is permanent and immediate.
    • Admin or moderator delete of other users' posts — out of scope for this feature; moderation is a separate concern.
    • Bulk delete from MyPostsScreen.
    • Undo delete.
